# 战斗过程和操作 UI 反馈音效设计

## 目标

为现有战斗播放和高频操作反馈增加短促、低音量的反馈音效，让玩家能听到掷骰、命中、治疗、护盾、异常状态、购买、刷新、放置、升级、出售、奖励领取、战斗开始和操作失败等关键反馈。

## 范围

- 只修改前端反馈表现，不修改服务端战斗结算、装备数值、遗物数值或平衡模型。
- 复用现有 `feedback.ts` 的战斗表现分类和 UI 反馈分类。
- 不新增外部音频文件，使用 Web Audio API 合成短音效，避免资源体积和授权问题。
- 复用现有顶部音乐开关作为音频总开关：关闭音乐时也关闭反馈音效。
- 浏览器音频权限未解锁时静默跳过，不显示错误，不阻塞操作。

## 前端行为

战斗播放中，`BattleView` 在当前战斗事件变化时，根据 `createBattlePresentation(event).kind` 映射音效：

- `roll`：短促掷骰音。
- `damage`：清晰命中音。
- `miss`：低沉抵消音。
- `heal`：上扬治疗音。
- `shield`：轻亮护盾音。
- `poison`、`weak`、`freeze`、`thorns`、`utility`：各自使用不同音色或节奏的短提示。

操作 UI 中，`pushUiFeedback` 创建 toast 时同步播放音效：

- 成功类：购买、放置、升级、出售、奖励领取、遗物领取、附魔完成。
- 信息类：刷新商店、战斗开始。
- 失败类：金币不足、放置失败、升级失败、通用操作失败。

音效音量低于背景音乐，单次时长控制在几百毫秒内。连续战斗事件按播放帧触发，音效本身短促，避免覆盖背景音乐。

## 测试

新增或扩展前端测试，覆盖：

- 战斗表现类型能映射到稳定的音效 cue。
- UI 反馈类型能映射到稳定的音效 cue。
- 音效合成器在禁用、无浏览器音频上下文、上下文被拒绝时不会抛错。
- `App` 将音效接入战斗事件变化和 UI toast 创建路径。

## 构建验证

本次修改影响战斗过程和前端表现，交付前必须运行 `npm run build`，确认 `dist-click\DogFight-standalone.cmd` 已由构建流程重新生成。
