# 休闲匹配降压与去扎堆设计

## 背景

当前休闲战斗匹配会在玩家点击匹配时写入自己的 `GhostSnapshot`，随后按同回合、胜负接近、`createdAt desc` 取第一条候选。这个规则会偏向刚刚创建的快照，导致玩家容易反复遇到同一批最近游玩的对手；同时候选允许 `wins <= 当前 wins + 1`，会匹配到更强记录的对手，难度压力偏高。

## 目标

- 休闲匹配优先选择比玩家低一胜场的对手：`targetWins = max(0, wins - 1)`。
- 候选池不再直接取最新快照，而是取一批候选后稳定打散，降低同一时刻玩家互相扎堆的问题。
- 排除当前 run 和当前用户自己的快照，避免玩家匹配到自己的历史或刚写入的快照。
- 找不到真实玩家快照时，离线对手也按低一胜场生成，保持降压方向一致。
- 保持前两回合训练对手规则不变，仍使用离线训练对手。

## 非目标

- 不新增数据库字段，不记录长期匹配历史。
- 不改多人 dogfight 房间内的配对规则。
- 不改战斗模拟、装备数值或奖励结算。

## 方案

在 `src/server/game/matchmaking.ts` 中新增纯函数 `selectCasualGhostSnapshot()`。API 层从数据库读取同回合、同用户排除、当前 run 排除、胜场在 `[targetWins, wins]` 范围内、败场接近的候选，交给该函数选择。

选择规则：

1. 优先 `wins === targetWins` 的候选。
2. 如果没有低一胜场候选，允许 `wins === 当前 wins` 的候选。
3. 同一优先级内用 `seed` 计算稳定分数排序，而不是按 `createdAt desc`。
4. `seed` 使用当前 run、回合、胜负和候选 id 组合，保证同一局面可复现，不依赖 `Math.random()`。

## 验收标准

- 有低一胜场候选时，匹配结果必须选择低一胜场候选，而不是同胜场或更高胜场。
- 选择不能只因为候选更新更晚就固定命中最新快照。
- API 匹配时排除当前用户自己的 ghost。
- 没有真实候选时，离线 fallback 的 `wins` 使用 `max(0, 当前 wins - 1)`。
- `npm run build` 成功，并刷新单文件版本。
