# 战斗状态点击说明 Tips 设计

## 背景

战斗界面已经把正面状态和负面状态显示为血条附近的状态芯片，例如护盾、荆棘、额外骰、中毒、虚弱、冻结和失效。当前芯片能提示状态存在和数量，但玩家仍需要记住每个状态的具体规则，尤其是在装备、职业道具和遗物叠加后，新玩家很难判断某个状态何时结算、影响什么数值、是否会消耗。

本设计目标是在不改变战斗规则和数值的前提下，让玩家点击状态芯片即可看到详细解释。

## 推荐方案

采用“复用现有浮层系统”的方案。

状态芯片点击后，在点击位置附近弹出与装备详情一致风格的 `floating-tip paper-card`。浮层显示状态名称、正面或负面分类、当前层数或数值、规则解释、结算时机和典型来源。点击外部区域关闭浮层；再次点击其他状态时，浮层内容切换到新状态。

这样可以保持战斗页、装备详情和遗物详情的交互一致，也能复用现有的浮层定位、视口边缘避让和移动端滚动能力。

## 交互规则

1. 战斗中的每个状态芯片都变为可点击控件，保留现有颜色、尺寸和正负面分行布局。
2. 点击状态芯片时，记录状态类型、显示文本、当前数值、所属阵营和点击锚点。
3. 浮层靠近被点击芯片展示，并沿用现有 tip 的视口边缘保护逻辑，避免超出屏幕。
4. 点击浮层外部关闭；战斗切换、离开战斗页或状态来源消失时关闭。
5. 键盘和读屏层面，状态芯片使用按钮语义，并提供 `aria-label`，例如“查看中毒说明”。

## 内容结构

状态说明由前端维护一张统一说明表，至少覆盖当前已经可见的状态：

- 护盾：说明会先吸收伤害，显示当前护盾值。
- 荆棘：说明受到直接伤害时会反弹，显示当前层数或反伤能力。
- 额外骰：说明会影响后续投骰或触发次数，显示剩余次数。
- 中毒：说明会在结算时造成持续伤害，显示层数和下一次毒伤提示。
- 虚弱：说明会降低后续造成的伤害，显示层数。
- 冻结：说明会限制或跳过行动，显示剩余时间或次数。
- 失效：说明会让装备或效果被跳过，显示剩余次数。
- 折叠入口 `+N`：如果现有 UI 有折叠状态，点击后应展示完整状态列表；列表中的每一项仍可打开对应说明。

说明文案应使用游戏内中文术语，不引入新的规则名称。无法从当前快照得到的数值不强行展示，只显示规则解释和已有可见数量。

## 前端结构

建议在 `src/App.tsx` 中新增轻量状态：

```ts
type StatusTipState = {
  key: string
  label: string
  value: number | string | null
  side: 'player' | 'opponent'
  polarity: 'positive' | 'negative'
  anchor: TipAnchor
}
```

现有状态芯片渲染逻辑增加 `onClick`，调用与装备 tip 类似的定位函数生成 `anchor`。新增 `StatusTip` 组件负责根据状态说明表渲染浮层内容，并复用 `.floating-tip.paper-card` 的基础样式。

如果当前状态芯片仍在 `BattleDog` 内部直接拼接，优先做局部提取：保留现有布局，只把“状态 chip 的文本、类型、点击处理”收束到小函数或小组件，避免重写整个战斗页。

## 样式

状态芯片在可点击后需要有轻量可发现性：

- 鼠标悬停时提高明度或阴影，但不改变芯片尺寸。
- 焦点态使用清晰描边，不能挤压血条布局。
- 浮层宽度沿用装备 tip 的移动端约束，避免遮挡过多战斗画面。
- 浮层内部使用紧凑标题、标签和说明段落，不做新的大卡片样式。

## 测试计划

先补前端结构测试，再实现行为：

1. `src/App.tooltip.test.ts` 增加用例，确认战斗状态芯片打开独立的状态说明 tip。
2. 确认状态说明表包含护盾、荆棘、中毒、虚弱、冻结、失效和额外骰。
3. 确认状态芯片包含按钮语义或点击处理，并有可访问名称。
4. 确认 tip 复用 `floating-tip paper-card`，而不是新增不一致的浮层样式。
5. 构建前运行相关测试；因为本次实现会影响前端展示和单文件可玩版本，最终必须运行 `npm run build` 并确认 `dist-click\DogFight-standalone.cmd` 重新生成。

## 非目标

- 不改变任何战斗规则、状态数值、装备数值或平衡模型。
- 不新增新的增益或减益类型。
- 不重做战斗页整体布局。
- 不更新外部数值 Excel；本需求只改变 UI 解释和交互，不影响数值模型。
