# 杀戮尖塔式探索地图 UI 与路线规则设计

## 背景

当前探索地图曾使用固定 12 层、每层 3 个节点的网格布局，并且每个节点默认连接下一层相邻 3 个位置。结果是连线密度过高、节点过多、可读性差，视觉上像一张半透明表格，而不是一个需要玩家做路线选择的地图。玩家也难以感受到“选定路线后只能沿着分支向前走”的压力。

本次修改的目标是明显重做地图界面，并同步调整路线规则，让地图更接近《杀戮尖塔》的随机前进结构。

## 目标

- 地图不再像固定表格，而是像一张纵向路线图。
- 每层节点数量有随机性，不再固定为 3 个。
- 路线只能向前推进，不能横向来回穿梭，也不能回头。
- 玩家每次只能选择当前路线后继节点。
- 任意一条从入口走到终点的完整路径，玩家对战数量应基本公平，稳定控制在 4 到 5 次。
- UI 要明显降低噪音：更少节点、更大节点、更清晰的连线、更强的当前可选状态。
- 已走路线、可选路线、未解锁路线必须一眼区分。
- 保留节点预览信息，包括战斗、怪物奖励、商店、事件、休息点。

## 非目标

- 不重做战斗、商店、事件、休息点的结算逻辑。
- 不修改装备、遗物、金币、伤害、治疗、护盾等数值。
- 不新增图片资源或装备图标。
- 不引入大型地图渲染库。

## 路线生成规则

地图改为 10 层，每层节点数为 2 到 4 个。每个节点获得稳定的横向槽位，用于前端布局。

生成原则：

- 同一个 run seed 生成结果必须稳定。
- 第 1 层开放 2 到 3 个入口，而不是全部节点固定开放。
- 每个非末层节点至少连接 1 个下一层节点，最多连接 2 个下一层节点。
- 连线只允许连接到下一层，不允许跳层、回头或连接同层节点。
- 下一层每个节点尽量至少有 1 个入边，避免生成永远不可达的孤岛。
- 连线优先连接横向距离近的节点，减少交叉和视觉混乱。
- 节点类型仍沿用现有类型：玩家对战、怪物战斗、固定商店、随机商店、装备商店、休息点、事件。

玩家对战节点需要额外做路径公平约束：

- 每条完整路径上的 `PLAYER_BATTLE` 节点数量必须为 4 或 5。
- 地图生成时优先按层控制玩家对战分布，例如在 10 层中选择 4 到 5 个“玩家对战候选层”。
- 玩家对战候选层不能只放在单一路线分支上，必须保证所有可达路径都会经过该层的玩家对战约束。
- 如果某层不是强制玩家对战层，可以生成怪物、商店、休息或事件节点，但不能让某条路径绕开过多玩家对战。
- 生成后需要做路径扫描；如果存在完整路径玩家对战数低于 4 或高于 5，则重新生成或修正该地图。

## 可达性规则

初始状态只开放地图入口节点。完成一个节点后，下一步只开放该节点的 `nextNodeIds`。

如果当前节点正在处理中，例如事件未处理、怪物掉落未领取、战斗未返回地图，则地图不开放任何新节点。完成当前节点后再显示下一步可选节点。

这样玩家不能从同层其他分支切换，也不能返回此前路线。

## UI 设计

地图界面要从“密集卡片网格”改成“路线板”：

- 需要明确评估路线方向。候选方向包括从上到下和从左到右，不能只沿用当前纵向布局。
- 从上到下的优点是接近《杀戮尖塔》玩家认知，层级推进直观，适合窄屏和移动端；缺点是层数较多时需要更高画布，容易需要滚动或缩放。
- 从左到右的优点是更适合 PC 宽屏，当前层、后续分支和右侧详情可以并排展示；缺点是移动端会更难承载，需要单独响应式布局。
- 默认推荐 PC 端使用从左到右的全屏路线板，移动端切换为从上到下或可横向滚动的简化路线板。
- 需要评估地图是否从普通页面区域升级为全屏呼出界面。由于当前地图同时承载路线、节点类型、奖励预览、当前战绩、金币、容错和待处理事件，普通嵌入式面板容易拥挤。
- 全屏呼出界面推荐作为默认方案：打开地图时覆盖主要游戏区域，顶部保留战绩和关闭/返回操作，中间展示路线，侧边或底部展示当前悬停/选中节点详情。
- 如果保留嵌入式地图，也必须显著减少节点内文字，并把长信息移到详情栏，否则会继续出现遮挡、重叠和阅读困难。
- 地图宽度收窄，中心区域作为主路线画布，避免横向铺满导致视觉发散。
- 节点从当前的长方形信息卡改成更紧凑的图章节点：上方图标、下方短标题，详细预览通过更短文案或悬浮信息承载。
- 节点尺寸减小但点击区域保持稳定，避免遮挡连线。
- 连线改为细实线或短虚线，默认灰色更淡；当前可选出口使用更醒目的蓝色或金色；已完成路线使用绿色或深棕色。
- 未解锁节点整体降低透明度，但不糊成一片；文字不再和背景混在一起。
- 当前可选节点要有清晰发光描边或抬起效果。
- 当前已选择但未完成的节点使用独立高亮，和“可选下一步”区分。
- 只显示少量必要文字：节点标题保留，长奖励预览不直接挤在地图节点里。
- 地图背景保留手绘纸张风格，但去掉过强的网格线和大面积半透明遮罩。

## 前端布局

前端不再根据固定 3 列数组计算位置，而是使用节点自带的 `x` 坐标或稳定派生坐标。布局方向需要支持 PC 与移动端不同承载方式。

推荐结构：

- `ExplorationMapNode` 增加可选 `x` 字段，取值为 0 到 1，表示节点在地图画布中的横向位置。
- 前端 `mapNodePosition()` 优先使用 `node.x`，旧存档没有 `x` 时回退到现有 `column` 逻辑。函数还需要支持 `horizontal` 与 `vertical` 两种方向。
- PC 端推荐全屏呼出层：左到右推进，层数映射到横向坐标，节点横向分布映射到纵向坐标；右侧固定详情栏展示选中节点信息、奖励预览和操作按钮。
- 移动端推荐上到下推进，或使用横向滚动路线板；节点只显示图标和短标题，详情进入底部抽屉，保证点击区域不被文字挤压。
- 地图呼出层需要有稳定的顶部栏，展示“第几张地图、第几层、胜场、容错、金币”，并保留明确的返回按钮。
- 当前事件、待领取掉落和节点详情不要塞进地图节点内部，应放在详情栏或底部详情区。
- 地图画布尺寸使用稳定约束，避免节点高亮、悬停、长标题导致布局跳动。

## 测试

需要更新地图生成测试：

- 同 seed 地图生成稳定。
- 地图为 10 层。
- 每层节点数在 2 到 4 个之间。
- 非末层节点至少 1 条出边，最多 2 条出边。
- 所有出边只连接下一层。
- 枚举或采样所有入口到末层的完整路径，玩家对战数量都必须在 4 到 5 次之间。
- 初始可选节点等于入口节点。
- 完成某节点后，只开放该节点后继节点。
- 地图包含战斗、商店、休息、事件等基础节点类型。

需要补充前端静态测试或样式断言：

- 地图节点使用新的路线板样式类。
- 不再依赖 3 列固定横坐标作为唯一布局来源。
- 锁定、可选、完成、当前节点状态具备不同样式。
- PC 地图使用全屏呼出层或等价的大画布承载方式，不再挤在普通窄面板里。
- 布局函数或样式支持横向推进与移动端纵向/滚动承载。
- 节点详情、奖励预览和待处理事件不直接撑大地图节点。

## 交付验证

本次修改影响游戏行为、前端展示和打包产物，交付前必须运行：

```bash
npm run build
```

本次不修改数值和平衡模型，因此不更新外部数值 Excel。
