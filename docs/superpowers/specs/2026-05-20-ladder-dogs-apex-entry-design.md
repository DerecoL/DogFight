# 天梯狗进入巅峰竞技场设计

## 背景

当前巅峰竞技场用于接收已经完成的跑局快照，并让该狗自动挑战巅峰榜。后端提交接口 `/api/apex/submit` 已经按“完成局”校验资格，但巅峰概览接口 `/api/apex` 只把 `mode: 'CASUAL'` 的完成局列入候选，因此天梯模式完成后的狗不会出现在可投入列表中。

用户目标是：天梯模式的狗也需要可以放到竞技场里面。

## 目标

- 休闲模式完成局和天梯模式完成局都能作为巅峰竞技场候选狗。
- 每个完成局仍然只能投入巅峰竞技场一次。
- 天梯结算、段位积分、巅峰挑战算法保持不变。
- 单文件可玩版本与主应用行为保持一致。

## 非目标

- 不新增单独的“天梯巅峰榜”。
- 不改变巅峰竞技场的排名、挑战顺序或种子狗生成逻辑。
- 不改变天梯结算公式、匹配公式、装备数值或遗物数值。

## 推荐方案

采用统一候选池：`/api/apex` 查询当前用户所有 `status: 'COMPLETE'` 且未提交过巅峰竞技场的跑局，不再限制 `mode: 'CASUAL'`。

提交接口 `/api/apex/submit` 保持“必须是自己的完成局、不能重复提交”的校验。由于它现有逻辑已经不限制模式，只需要用测试锁定天梯完成局可提交的行为。

前端 `ApexArena` 的候选文案从“先在休闲模式完成一局”调整为“先在休闲或天梯完成一局”，大厅里巅峰模式描述也同步避免暗示只有休闲狗可进入。

## 数据流

1. 玩家完成休闲或天梯跑局。
2. `/api/apex` 读取当前用户已提交过的 `sourceRunId`。
3. `/api/apex` 返回所有未提交过的完成局，包含 `CASUAL` 与 `LADDER`。
4. 前端候选列表直接渲染返回的 `Run[]`。
5. 玩家提交某个候选狗后，`/api/apex/submit` 用该跑局快照生成 `ApexEntry`。
6. 该跑局再次被 `/api/apex` 排除，避免重复提交。

## 测试计划

- 新增或更新服务端 API 测试：
  - 构造一个完成的休闲局和一个完成的天梯局。
  - 调用 `/api/apex`，确认两个完成局都在候选列表中。
  - 提交天梯局到 `/api/apex/submit`，确认返回玩家巅峰条目。
  - 再次调用 `/api/apex`，确认已提交的天梯局不再出现在候选列表。
- 更新前端结构测试，确认巅峰文案不再写死为“休闲模式完成一局”。
- 运行 `npm test` 验证回归。
- 因为本次会影响游戏入口行为和单文件包逻辑，最终实现后必须运行 `npm run build`，确认 `dist-click\DogFight-standalone.cmd` 同步成功。

## 数值文档影响

本次不修改装备数值、遗物数值、职业道具数值、战斗参数、经济价格或平衡模型，因此不需要更新 `C:\Users\User\Desktop\狗骰乱斗\狗骰乱斗.xlsx`。

## 风险与边界

- 天梯狗通常有天梯积分结算，但投入巅峰竞技场不应再次触发天梯结算。
- 已经提交过巅峰竞技场的天梯局必须继续通过 `sourceRunId` 唯一约束阻止重复提交。
- 单文件版本有独立内联 API 模拟逻辑，主应用改动完成后要通过构建脚本同步，不能只改源码。
