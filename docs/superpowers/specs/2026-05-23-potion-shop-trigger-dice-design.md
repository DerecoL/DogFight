# 药水商店与触发骰子修改设计

## 背景

当前游戏已有普通装备商店、遗物三选一、升级三选一和附魔选择。装备触发骰子主要来自道具定义 `ItemDef.dice`，战斗时再叠加附魔额外点数、遗物偏移、点金手映射、职业特殊规则等效果。

本次新增“药水商店”，用于永久修改某一件非职业装备实例的基础触发骰子值。药水商店本身也是三选一奖励商店，玩家进入后必须能看到每瓶药水的具体数值，再决定选择哪一瓶以及用于哪件装备。

## 目标

- 在回合三选一中新增药水商店入口。
- 药水商店出现概率与遗物、升级一致，并且独立判定。
- 药水商店提供 3 个药水选项，玩家从中选择 1 个。
- 药水效果可修改普通装备实例的基础触发骰子值。
- 职业装备不可使用药水，前端禁用并由后端强校验。
- 药水修改后的基础触发骰子仍继续被其他道具和遗物影响。
- 单文件可玩版本在实现完成后必须通过 `npm run build` 同步更新。

## 非目标

- 不调整药水以外的装备数值、遗物数值、职业装备数值或经济价格。
- 不把药水做成普通可购买商品，不进入 5 件商品货架。
- 不允许药水为同一骰点制造“一次掷骰多次触发”的效果。
- 不改变职业装备本身的触发规则。

## 回合入口规则

新增商店类型 `POTION`。

在第 3 回合及以后生成三选一时，保留现有普通商店池先生成 3 个不重复选项，然后分别独立尝试插入特殊商店：

- 遗物商店：现有概率不变。
- 升级商店：现有概率不变，仍要求存在可升级装备。
- 药水商店：与遗物、升级相同概率，并独立判定。

药水、遗物、升级可以在同一组三选一里同时出现。若特殊商店被判定出现，则替换一个当前普通商店选项；不替换已经出现的遗物、升级或药水选项。

## 药水选项生成

进入药水商店后生成 3 个药水选项。每个选项先按权重抽类别，再按类别抽具体数值。

类别权重：

| 类别 | 权重 | 效果 |
| --- | ---: | --- |
| 类别1 | 50 | 增加 1 个随机点数触发 |
| 类别2 | 10 | 增加 2 个随机点数触发 |
| 类别3 | 10 | 额外增加 1 个随机点数触发 |
| 类别4 | 10 | 修改为小点、大点或极值触发 |
| 类别5 | 1 | 修改为全点数触发 |

具体规则：

- 类别1：从 `1/2/3/4/5/6` 随机 1 个点数，并入目标装备基础触发骰子集合。
- 类别2：从 `1/2/3/4/5/6` 随机 2 个不同点数，并入目标装备基础触发骰子集合。
- 类别3：从 `1/2/3/4/5/6` 随机 1 个点数，并入目标装备基础触发骰子集合；文案强调“额外增加”，用于表达这是强化特定装备的稀有药水。
- 类别4：从“小点 `1/2/3`”、“大点 `4/5/6`”、“极值 `1/6`”中随机 1 个，替换目标装备基础触发骰子集合。
- 类别5：替换为全点数 `1/2/3/4/5/6`。

所有选项在生成时就固定具体值，前端卡片必须显示完整文本，例如“增加 2/5 点触发”或“改为极值 1/6 触发”。

## 基础触发骰子语义

药水修改的是某一件装备实例的基础触发骰子，不是最终触发结果。

最终触发判断流程：

1. 读取装备定义的默认 `dice`。
2. 若该装备实例存在药水基础骰子覆盖，则使用覆盖后的基础骰子。
3. 对基础骰子应用附魔额外触发点数。
4. 对结果应用遗物偏移，例如胡萝卜 `+1`、纸巾 `-1`。
5. 再走现有特殊触发规则，例如点金手映射、柴犬按容量触发、皇帝相邻改天命等。

因此，药水改出来的点数还会继续被其他道具和遗物影响。

重复点数只保留一次。例如装备已有 `1` 点触发，再使用“增加 1 点触发”的药水，最终基础集合仍只包含一个 `1`，不会让一次 `1` 点掷骰触发两次。

## 数据模型

在道具实例上新增可选字段保存药水修改后的基础触发骰子集合。

建议字段：

- Prisma `ItemInstance.triggerDiceOverride String?`
- 运行时 `GameItem.triggerDiceOverride?: number[] | null`

字段含义：

- `null` 或空值：使用装备定义默认 `dice`。
- 非空数组：使用该数组作为装备实例基础触发骰子。

写入时必须归一化为 `1-6` 的去重升序数组。

## 阶段与 API

新增阶段 `POTION_CHOICE`。

三选一选择 `POTION`：

- 校验当前在 `CHOICE`。
- 生成 3 个药水选项。
- 将阶段更新为 `POTION_CHOICE`。
- 清空普通商店商品。

新增选择接口：

- `POST /api/runs/:runId/potion/select`
- 请求体：`{ potionId: string, itemId: string }`

后端校验：

- 当前阶段必须是 `POTION_CHOICE`。
- `potionId` 必须存在于当前药水三选一。
- `itemId` 必须属于当前跑局。
- 目标装备不能是职业装备。
- 目标装备必须是可触发装备或可被触发规则影响的普通装备；默认允许普通装备，即使当前默认 `dice` 为空，也可被类别4/5设为可触发。

应用成功后进入 `PREP`。

## 前端交互

药水商店页面复用现有奖励选择面板风格：

- 上方展示 3 张药水卡片。
- 每张卡片显示类别文案和具体点数。
- 下方让玩家选择目标装备。
- 职业装备显示为不可用状态，并提示“职业装备不可使用药水”。
- 当前选中的药水和目标装备都明确高亮。
- 提交按钮在未选药水或未选有效装备时禁用。

道具卡片、商店预览、悬浮提示、战斗快照都应显示药水后的最终触发点数显示，并继续体现遗物偏移结果。

## 多人狗斗与自动选择

狗斗模式中若玩家超时停留在 `POTION_CHOICE`：

- 自动选择第 1 个药水。
- 自动应用到第 1 件非职业装备。
- 若没有合法目标，直接进入 `PREP`。

## 测试

新增或更新测试覆盖：

- 药水类别按权重生成，三选一选项包含具体值。
- 药水商店与遗物、升级独立出现。
- 类别1/2/3 合并基础触发点数且去重。
- 类别4/5 替换基础触发点数。
- 药水修改后的基础点数会继续被遗物偏移影响。
- 职业装备不能使用药水。
- API 成功应用药水后进入 `PREP`。
- 狗斗自动流程能处理 `POTION_CHOICE`。
