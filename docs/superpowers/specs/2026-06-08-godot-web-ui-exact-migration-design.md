# Godot 版 Web UI 完全复刻迁移规格

## 背景

当前 `godot-client` 已经实现一套 Godot 原生工作台式 UI：`RunScreen.gd` 以顶部工具条、左侧标签和右侧滚动内容承载大厅、跑局、账号、成就、每日、商城、排行、赛季、房间和设置等入口。这套结构来自此前“Godot 原生重设计 + Web 功能对齐”的方向。

用户已明确要求：迁入 Godot 的版本中，UI 的交互逻辑、样式和结构都必须与现有 Web 版完全一样。因此，本规格取代 `docs/superpowers/specs/2026-06-02-godot-native-client-redesign.md` 中“Godot 原生体验重设计”的方向。Godot 客户端仍然可以使用 Godot 原生控件、场景树和资源系统实现，但玩家看到的屏幕结构、视觉层级、交互入口、流程顺序和控件行为必须以 Web 版为准。

## 目标

- 将 Web 版 `src/App.tsx` 和相关 CSS 中的 UI 结构完整迁入 Godot。
- Godot 版的屏幕拆分、流程入口、弹窗、浮层、装备格、商店货架、地图、奖励选择、房间和战斗回放必须与 Web 版一一对应。
- Godot 版的视觉样式必须复刻 Web 版的手绘纸片、卡片、按钮、资源胶囊、装备格、遗物栏、地图节点、弹窗、战斗 HUD 和结算面板。
- Godot 版的交互逻辑必须复刻 Web 版，包括点击查看详情、拖拽/点击放置、商店购买确认、出售区、奖励选择、地图节点预览、房间轮询状态和战斗回放控制。
- 服务端仍是唯一权威状态来源。Godot 只负责展示、输入、请求 API、播放服务端返回的战斗事件。

## 非目标

- 不继续维护“Godot 工作台式 UI”作为正式迁移目标。
- 不把 Web UI 简化成左侧标签页和文本按钮列表。
- 不为了 Godot 原生便利而改变 Web 版屏幕流程。
- 不迁移 TypeScript 战斗、商店、地图、赛季、成就、每日任务或多人房间规则到 GDScript。
- 不用 Godot 内嵌 Web 页面作为默认方案；除非后续单独确认，否则 Godot 必须原生实现 UI。

## 当前问题根因

当前 Godot UI 与 Web UI 不一致不是单个样式缺失，而是迁移架构方向不同：

- Web 版按玩家流程拆屏：登录、昵称、模式大厅、跑局地图、奖励选择、商店货架、装备板、战斗回放、结算、账号商城、成就每日、排行赛季、多人房间。
- Godot 版目前把多数功能集中在 `RunScreen.gd` 单屏工作台，通过 `current_tab` 切换内容。
- Web 版大量交互依赖专用组件和浮层，例如 `ExplorationMapView`、`ShopShelf`、`InventoryBoard`、`GridPanel`、`RelicRail`、`FloatingTip`、奖励选择面板、地图节点详情、房间详情和战斗回放面板。
- Godot 版目前大量条目是纯按钮和文本行，缺少 Web 版相同的空间结构、卡片层级、图标/贴纸资产呈现、浮层提示和状态反馈。

因此修复方式必须是重建 Godot UI 架构，而不是在当前工作台 UI 上局部调样式。

## Source of Truth

Godot 迁移以以下文件为准：

- `src/App.tsx`：屏幕流程、组件结构、状态分支、交互行为。
- `src/App.css` 及相关 UI primitive 样式：视觉层级、卡片/按钮/装备格/地图/弹窗样式。
- `public/assets`：Web 版正在使用的狗、装备、遗物、地图、背景和贴纸资源。
- 服务端 API 路由：状态读写仍通过现有 Fastify API。

Godot 中新增或重写的 UI 文件不得自行发明新的玩家流程。如果 Web 与 Godot 有差异，默认 Web 正确。

## 屏幕结构迁移

Godot 主场景必须从当前三屏结构扩展为与 Web 流程一致的屏幕集合：

- `LoginScreen`：账号、密码、注册、TapTap 调试提示、快速开始提示。
- `NicknameSetupScreen`：昵称设置流程。
- `ModeLobbyScreen`：模式大厅、当前账号摘要、继续跑局、休闲/天梯/多人/巅峰入口。
- `RunShellScreen`：跑局全局顶栏、资源状态、音乐、返回大厅、退出。
- `ExplorationMapScreen`：地图路线、节点详情、当前事件、待领取掉落、路线草稿工具。
- `RunShopScreen`：商店货架、刷新、购买详情、出售区、匹配入口。
- `InventoryBoardPanel`：装备栏、背包、遗物栏，与 Web 的 `InventoryBoard` 对齐。
- `RewardChoiceScreen`：商店三选一、职业装备、附魔、遗物、升级、药水选择。
- `BattleReplayScreen`：双方快照、血条、骰子、装备触发、日志、速度、跳过、结算。
- `RunSettlementScreen`：跑局结束、奖励、天梯结算、返回大厅。
- `AccountShopScreen`：账号商城、外观购买、装备状态。
- `AchievementsScreen`：成就分类、每日任务、领取、刷新。
- `LeaderboardsScreen`：天梯榜、巅峰总榜、巅峰日榜、配置快照。
- `SeasonScreen`：当前赛季、赛季历史、赛季快照。
- `DogfightRoomsScreen`：房间列表、创建、加入、匹配。
- `DogfightRoomDetailScreen`：玩家席位、阶段状态、选狗、准备、战斗观战、房间战报。
- `AccountSettingsScreen`：外观装备、音乐设置、个人设置。

`RunScreen.gd` 不得继续作为所有功能的大容器。正式迁移实现必须把其中的大厅、跑局、账号、成就、每日、商城、排行、赛季、房间和设置逻辑拆入对应屏幕或组件；旧工作台入口只能作为迁移期间的临时兼容层，不能作为验收目标。

## 交互逻辑迁移

### 装备与背包

- 保留 Web 的装备栏和背包单行格结构。
- 装备栏槽位数量必须随遗物效果变化。
- 物品可以点击选中并显示浮层详情。
- 桌面端必须支持拖拽到装备栏、背包和出售区。
- 移动端必须支持点击物品再点击目标格的等效操作。
- 放置、出售、升级、药水、附魔等操作都必须调用 API，并以 API 返回的 run 刷新本地状态。

### 商店

- 商店必须是 Web 版货架结构，而不是按钮列表。
- 商品卡必须展示品质、已拥有数量、画芯、占格、触发点数、效果、价格和折扣。
- 点击商品先打开详情/浮层，再确认购买。
- 刷新按钮、出售区和匹配按钮位置必须与 Web 版一致。
- 购买后进入背包，不得在 Godot 本地推断结果。

### 地图

- 地图必须复刻 Web 的路线板：层级标记、连线、当前/可选/已完成节点状态、节点预览、节点详情面板。
- 地图节点必须支持预览、选择、进入、查看掉落、查看怪物装备。
- 路线草稿工具必须包含查看、画笔、橡皮、清空，层级不能遮挡节点和详情。

### 奖励选择

- `CHOICE`、`CLASS_REWARD`、`ENCHANT_CHOICE`、`RELIC_CHOICE`、`UPGRADE_CHOICE`、`POTION_CHOICE` 必须拆成与 Web 相同的选择面板。
- 职业装备和附魔出现时必须保留 Web 的仪式屏和点击继续逻辑。
- 药水和附魔选择后，后续点击装备的行为必须与 Web 相同。

### 战斗回放

- Godot 不重新模拟战斗，只播放服务端 `BattleResult.events`。
- 战斗 UI 必须与 Web 的双方区域、血条、骰子、装备触发、高亮、日志、速度、跳过和完成结算对齐。
- 特效层不得遮挡 HUD、日志、按钮、结算操作。

### 多人房间

- 房间列表、房间详情、玩家席位、房主、准备、选狗、阶段倒计时、观战和战报必须与 Web 流程一致。
- 房间轮询必须绑定当前房间和屏幕生命周期，离开房间或切屏后停止旧轮询。

## 样式迁移

Godot UI Kit 必须以 Web 样式为模板，而不是重新定义一套工作台风格。

必须复刻：

- 手绘纸片卡片、木纹/纸质边框、手绘按钮。
- 顶栏资源胶囊和账号头像/称号。
- 模式大厅狗狗卡片和模式卡片。
- 商店商品卡、装备画芯、品质标签、价格标签。
- 装备格、背包格、拖拽高亮、选中态、可升级态。
- 遗物 6 槽轨道、遗物详情浮层。
- 地图节点贴纸、路线连线、地图详情侧栏、路线草稿工具。
- 奖励选择卡、职业/附魔仪式屏、结算卡。
- 弹窗、浮层提示、Toast、阻塞确认层。

尺寸必须稳定。按钮、卡片、装备格、弹窗、列表项、地图节点、战斗 HUD 不得随内容长度任意撑开关键布局。长文本必须换行、截断或内部滚动。

## 层级规则

Godot 版必须显式规划 CanvasLayer 或等效层级：

1. 背景与装饰层。
2. 主屏幕内容层。
3. 拖拽与地图草稿层。
4. 战斗特效层。
5. 浮层提示层。
6. Toast 层。
7. 普通弹窗层。
8. 阻塞确认层。
9. 全局 loading / 断线重连层。

每次新增 UI、拖拽、特效、地图、弹窗或提示时，都必须检查不会被背景、卡片、画布、节点或后绘制元素压住。

## 资源与性能

- 优先复用 Web 版已有 WebP 和贴纸资源。
- Godot 导入前检查文件尺寸、体积、格式、压缩和加载时机。
- 首屏、大厅、地图、商店、战斗和装备图标不得一次性加载无关大图。
- 装备和遗物图标继续使用统一透明底手绘贴纸风格。
- 若 Godot 需要格式转换，必须用可重复脚本生成，不允许手工临时替换。

## 测试策略

### 结构测试

- Godot smoke 测试检查每个 Web 对应屏幕存在。
- 检查 `RunScreen.gd` 不再承载所有主流程。
- 检查每个 run phase 映射到正确屏幕或面板。

### 交互测试

- 装备点击详情、拖拽放置、点击目标格放置、出售、升级、药水、附魔。
- 商店商品详情、购买、刷新、出售区、匹配。
- 地图节点预览、进入、事件处理、奖励领取/放弃、怪物装备预览。
- 奖励选择、仪式屏继续、跳过升级。
- 房间创建、加入、离开、准备、选狗、战斗观战。

### 样式和层级测试

- 桌面和移动尺寸下检查核心容器稳定。
- 检查 Toast、浮层、弹窗、拖拽层、地图工具、战斗特效不遮挡关键按钮。
- 检查长文本、空状态、加载态、禁用态、选中态不会造成布局跳动。

### 构建测试

- 修改 Godot UI 后运行相关 Godot headless smoke。
- 若修改会影响游戏行为、前端展示、资源、数据、战斗逻辑或打包逻辑，必须运行 `npm run build`，确保单文件可玩版本同步生成。
- 若只修改本规格文档，可不运行 `npm run build`，但最终回复必须说明原因。

## 实施阶段

### 阶段一：迁移基座重置

- 建立 Web 屏幕到 Godot 场景的映射。
- 拆分 `RunScreen.gd`，保留 API 和 Store 能力。
- 建立与 Web 样式对应的 Godot UI Kit。
- 建立统一浮层、弹窗、Toast、拖拽和 loading 层。

### 阶段二：核心跑局流程

- 复刻模式大厅、跑局顶栏、商店、装备板、奖励选择、地图。
- 优先确保玩家能按 Web 流程完成一局。
- 保留桌面拖拽和移动点击目标格两套交互。

### 阶段三：战斗与结算

- 复刻战斗回放结构、装备触发、高亮、日志、速度控制和结算。
- 检查特效层级和性能。

### 阶段四：账号与长期系统

- 复刻账号商城、外观装备、成就、每日任务、排行、赛季、历史快照。

### 阶段五：多人房间

- 复刻房间列表、房间详情、玩家席位、阶段轮询、选狗、准备和观战。

### 阶段六：视觉验收与性能收口

- 对照 Web 版逐屏检查样式、结构、交互。
- 补齐 Godot smoke 和层级回归。
- 检查资源体积、加载速度和移动端尺寸稳定。

## 验收标准

Godot 迁移版只有满足以下条件才算完成：

- 玩家从登录到完成一局的主流程，与 Web 版屏幕顺序和交互一致。
- 玩家看到的 UI 结构与 Web 版一致，不再是工作台式标签页总览。
- 商店、装备、背包、遗物、地图、奖励、战斗、结算、房间等核心区域视觉样式与 Web 版一致。
- 所有改变状态的操作仍由服务端 API 决定结果。
- 桌面和移动尺寸下关键 UI 不重叠、不抖动、不被遮挡。
- Godot headless smoke、相关 UI 回归和必要构建通过。
