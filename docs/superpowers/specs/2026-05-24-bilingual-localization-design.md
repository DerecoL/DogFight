# 中英文全量本地化设计

## 背景

当前游戏以中文为唯一展示语言。界面文案主要集中在 `src/App.tsx`，装备、遗物、狗狗特性、效果描述和部分战斗文本来自 `src/server/game/data.ts`、`src/server/game/battle.ts`、`src/server/game/enchant.ts`、`src/shared/rule-terms.ts` 和服务端 API 错误消息。

本次目标是让玩家可以在中文和英文之间切换，并让英文模式覆盖完整可玩体验，而不是只翻译入口按钮或少量标题。

## 推荐方案

采用“集中翻译表，保留现有数据结构”的方案。

保留现有中文数据作为基准数据，不把 `ItemDef`、`RelicDef`、`DOGS` 等核心数据结构改成双语字段。新增前端本地化模块，按稳定 ID、枚举值、效果类型和已知消息 key 提供英文映射。组件渲染时通过当前语言读取展示文本。

这样可以把变更集中在展示层和少量文本生成函数中，降低对服务端状态、存档结构、API 合约和既有测试的影响。

## 目标

- 支持 `zh-CN` 和 `en-US` 两种语言。
- 默认语言为中文。
- 玩家可以在界面中切换语言，选择持久化到 `localStorage`。
- 切换语言后当前页面即时更新，不要求重新开始跑局。
- 英文模式覆盖主界面、按钮、模式、教程、装备、遗物、职业道具、规则术语、状态说明、战斗日志、服务端错误提示和常见反馈消息。
- 不改变任何装备数值、遗物数值、触发点、品质倍率、战斗公式、经济价格或匹配规则。
- 实现完成后运行 `npm run build`，同步生成单文件可玩版本。

## 非目标

- 不引入第三方 i18n 框架。
- 不改变数据库 schema。
- 不改变 API 返回的核心业务字段。
- 不将中文从源码中彻底删除。
- 不用机器翻译式的运行时字符串替换作为主要方案。
- 不更新外部数值 Excel，因为本次只改变语言和展示文案，不改变数值或平衡模型。

## 用户体验

语言入口放在玩家能稳定看到的位置，优先使用顶部栏或设置区域的小型分段控件：

- `中文`
- `English`

切换控件应保持轻量，不占用主要游戏操作区域。英文模式下控件展示 `中文 / English`，让玩家可以辨认返回中文的入口。

语言选择保存到浏览器本地：

```text
localStorage key: dogfight-language
values: zh-CN | en-US
```

如果本地值不存在或无效，则使用 `zh-CN`。

## 本地化模块结构

建议新增 `src/i18n` 目录，至少包含：

- `types.ts`：语言类型、翻译 key 类型、辅助类型。
- `dictionary.ts`：通用界面文案、按钮、模式、状态、错误消息。
- `game-text.ts`：装备、遗物、狗狗、规则术语、附魔、药水、品质等游戏文本。
- `battle-text.ts`：战斗日志、战斗状态、战斗展示文案。
- `LanguageProvider.tsx`：React context、当前语言、切换函数和持久化逻辑。

`App` 入口由 `LanguageProvider` 包裹。组件通过 `useLanguage()` 或 `useT()` 读取当前语言和翻译函数。

## 数据映射策略

装备、遗物和职业道具使用稳定 ID 映射英文：

```ts
itemTextById['dog-house'] = {
  name: 'Puppy Kennel',
  description: 'Restore health and steal one enemy buff, prioritizing Thorns before Haste.',
}
```

英文渲染时：

1. 优先按 `def.id` 查询英文翻译。
2. 如果缺少翻译，回退到原中文字段。
3. 测试应覆盖“所有现有装备和遗物 ID 在英文表里有映射”，避免长期漏翻。

狗狗、品质、商店类型、模式、天梯段位、状态名称、规则术语等使用枚举或 term key 映射。

## 动态效果文案

当前项目存在大量根据品质、数值和特殊效果动态拼接的中文文案。英文模式不做简单字符串替换，而是为关键生成函数增加语言参数或新增英文生成函数。

需要覆盖的重点包括：

- `qualityLabel`
- `dogNames` 和 `dogTraits`
- `shopTypeLabel`、`shopTypeDescriptions`
- `effectText`
- 装备详情中的触发点、尺寸、价格、出售、已拥有提示
- 附魔说明
- 药水说明
- 遗物效果说明
- 状态 tip 说明
- 战斗事件文本

无法一次结构化的中文动态文本允许使用“已知中文消息映射”作为过渡，但实现时必须为测试覆盖的现有战斗日志提供英文输出。

## 服务端错误提示

服务端可以继续返回中文 `error` 字段，避免改变 API 合约。前端展示错误时先通过已知错误文案表转换：

```ts
localizeServerError(errorText, language)
```

英文表覆盖当前服务端常见错误，例如：

- 无效狗狗选择
- 当前不能调整装备
- 当前没有可选遗物
- 装备不存在
- 职业装备不可使用药水
- 当前没有待结算战斗
- 斗狗房间加载失败
- 斗狗房间操作失败

如果错误未命中映射，英文模式回退显示服务端原文，避免吞掉错误。

## 测试设计

先写失败测试，再实现。

建议新增或更新以下测试：

- `src/i18n/dictionary.test.ts`：验证语言持久化、无效语言回退、核心翻译 key 存在。
- `src/i18n/game-text.test.ts`：验证所有 `ITEM_DEFS`、`RELIC_DEFS`、`DOGS` 在英文模式下有名称和描述映射。
- `src/App.structure.test.ts`：验证 `LanguageProvider`、语言切换控件和 `localStorage` key 存在。
- `src/App.effect-text.test.ts`：验证关键装备效果、附魔、药水、状态说明在英文模式下不再输出中文术语。
- `src/feedback.test.ts` 或新增战斗文本测试：验证典型战斗日志可输出英文。

实现后运行：

```bash
npm test
npm run build
```

## 构建与交付

本次实现会影响前端展示、游戏文案、战斗日志和单文件可玩版本，因此完成代码修改后必须运行：

```bash
npm run build
```

构建成功后确认单文件版本已同步生成：

```text
E:\AI-GPT\DogFight\dist-click\DogFight-standalone.cmd
```

本次不更新：

```text
C:\Users\User\Desktop\狗骰乱斗\狗骰乱斗.xlsx
```

原因：只改变语言展示和文案映射，不改变装备数值、遗物数值、职业道具数值、战斗参数、触发点、品质倍率、治疗、护盾、伤害、控制或经济价格。

## 风险与处理

- 文案体量大：通过测试强制覆盖所有稳定 ID，避免漏翻装备和遗物。
- 动态日志复杂：优先重构常见文本生成函数，无法立即结构化的部分用已知消息映射兜底。
- 结构测试依赖源码字符串：更新测试时改为检查本地化结构和翻译覆盖，不再依赖单一中文硬编码。
- 英文文本较长：语言控件和关键按钮需要 CSS 验证，避免英文溢出。

## 实施顺序

1. 建立 i18n 类型、字典和测试。
2. 接入 `LanguageProvider` 和语言切换控件。
3. 本地化主界面和通用组件文案。
4. 本地化装备、遗物、狗狗、规则术语和动态效果文本。
5. 本地化战斗日志、反馈消息和服务端错误展示。
6. 跑测试、修复英文布局问题并执行构建。
