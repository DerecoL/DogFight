# DogFight Figma UI 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**目标：** 在 Figma 中为 DogFight 搭建一套完整的“手绘绘本桌游”风格 UI，覆盖登录、狗狗图鉴、商店选择、商店工作台、匹配确认、战斗回放和结算。

**架构：** 先建立 foundations 和可复用组件，再组合 PC 高保真屏幕。所有屏幕共享顶部状态条、章节进度和手绘组件语言；选狗页使用可扩展角色库结构，避免固定三狗布局。

**技术栈：** Figma Design 文件、Figma Plugin API / Figma MCP、React 项目现有 UI 语义作为信息来源。

## Figma 文件结构

实施完成后，Figma 文件应包含这些页面：

- `01 Foundations`
- `02 Components`
- `03 Screens - PC`
- `04 Responsive Notes`

- `按钮 / Primary Button`
- `狗狗卡 / Dog Card`

## 任务 1：创建 Foundations 页面

**Figma 页面：**

- 创建：`01 Foundations`

**产出：**

- 色彩样板
- 字体层级
- 描边、阴影、圆角、间距规则
- 手绘风格说明小样

- [ ] **Step 1：创建页面和基础分区**

  在 Figma 中创建页面 `01 Foundations`，并放置 5 个横向分区：

  1. `颜色 / Colors`
  2. `字体 / Typography`
  3. `描边与阴影 / Strokes & Shadows`
  4. `间距与圆角 / Spacing & Radius`
  5. `手绘风格规则 / Hand-drawn Rules`

  验收：页面顶部可以一眼看到 5 个清晰标题，分区之间间距一致。

- [ ] **Step 2：建立颜色样板**

  创建以下颜色样板，每个样板包含色块、中文名、用途和 hex 值：

  - `纸张 / Paper`：`#FFF7E8`
  - `浅纸张 / Paper Light`：`#FFFCF4`
  - `墨棕 / Ink Brown`：`#3B2B22`
  - `木桌 / Table Brown`：`#7B5C40`
  - `主蓝 / Action Blue`：`#5D8CFF`
  - `橘红 / Brand Coral`：`#FF8B61`
  - `金币黄 / Coin Yellow`：`#FFE082`
  - `治疗绿 / Heal Mint`：`#B8F2D0`
  - `危险粉 / Danger Pink`：`#FF9DB7`
  - `装备蓝 / Utility Sky`：`#89D9FF`
  - `稀有紫 / Rare Lavender`：`#D5B8FF`

  验收：色板不应呈现单一色系；暖色、冷色、功能色都有明确区分。

- [ ] **Step 3：建立字体层级**

  创建字体样张：

  - `游戏标题`：40px，粗体，深棕，用于封面和大章节标题。
  - `章节标题`：28px，粗体，深棕，用于每个屏幕主标题。
  - `区块标题`：20px，粗体，深棕，用于卡片和面板标题。
  - `正文`：15px，常规，深棕，用于说明文案。
  - `辅助说明`：13px，常规，灰棕，用于标签和弱提示。
  - `数值徽章`：14px，粗体，深棕，用于金币、胜场、生命等。

  验收：正文和辅助说明在纸张背景上清晰可读；手绘感只通过形状和装饰表达，不牺牲文字清晰度。

- [ ] **Step 4：建立手绘描边和阴影规则**

  创建 4 个示例卡片：

  - 普通卡片：3px 墨棕描边，16px 左右不规则圆角，4px x 5px 纸片阴影。
  - 选中卡片：4px 主蓝外描边，轻微上移。
  - 禁用卡片：降低透明度，保留描边，不只依靠灰色。
  - 危险卡片：粉红底色，墨棕描边，危险图标或文字提示。

  验收：4 个状态无需阅读说明也能区分。

- [ ] **Step 5：记录移动端适配原则**

  在 `04 Responsive Notes` 页面先创建一个文字区，记录：

  - PC 是主设计目标。
  - 移动端多栏改纵向。
  - 装备栏和背包格横向滚动。
  - 主按钮在窄屏下靠近底部。

  验收：移动端范围清楚标注为说明级，不误导为完整移动高保真产出。

## 任务 2：创建核心组件库

**Figma 页面：**

- 创建或修改：`02 Components`

**产出：**

- 按钮、输入、状态条、章节进度、徽章、卡片、格子、日志等组件。

- [ ] **Step 1：创建按钮组件**

  创建 4 类按钮：

  - `按钮 / Primary Button`：主蓝底、墨棕描边、白色文字，用于开始、匹配、继续。
  - `按钮 / Secondary Button`：浅纸张底、墨棕描边、深棕文字，用于返回或次要操作。
  - `按钮 / Danger Button`：危险粉底、墨棕描边，用于出售。
  - `按钮 / Icon Button`：方形纸片按钮，用于退出、关闭、刷新。

  每类按钮至少包含默认、悬停、禁用状态。

  验收：按钮尺寸稳定，中文按钮文案不会溢出。

- [ ] **Step 2：创建输入框组件**

  创建 `输入框 / Text Input`：

  - 默认状态
  - 聚焦状态
  - 错误状态

  输入框需要有标签、输入内容、错误提示槽位。

  验收：错误状态不能只靠红色，需要显示错误文案区域。

- [ ] **Step 3：创建顶部状态条**

  创建 `顶部状态条 / Top Status Bar`，包含：

  - 游戏标识
  - 当前账号
  - 胜场徽章
  - 容错/失败次数徽章
  - 金币徽章
  - 回合徽章
  - 退出按钮

  验收：在 1440px 画板宽度下不拥挤；账号较长时有截断或换行策略。

- [ ] **Step 4：创建章节进度组件**

  创建 `章节进度 / Chapter Progress`，包含 6 个阶段：

  - 登录
  - 选狗
  - 商店选择
  - 商店
  - 匹配
  - 战斗

  每个阶段至少有未完成、当前、已完成三种状态。

  验收：当前阶段在远距离缩放下仍明显。

- [ ] **Step 5：创建狗狗卡和狗狗详情组件**

  创建：

  - `狗狗卡 / Dog Card`
  - `狗狗详情 / Dog Detail Panel`

  狗狗卡内容：

  - 手绘头像占位
  - 名字
  - 特性摘要
  - 标签区

  详情面板内容：

  - 大头像
  - 名字
  - 被动特性
  - 策略说明
  - 难度或推荐标签
  - 开始按钮

  验收：狗狗卡可以复制扩展到 12 张以上，不破坏选狗页结构。

- [ ] **Step 6：创建商店和物品组件**

  创建：

  - `商店类型卡 / Shop Type Card`
  - `商店物品卡 / Shop Item Card`
  - `物品详情 / Item Detail Panel`
  - `价格徽章 / Price Badge`
  - `骰点标签 / Dice Trigger Badge`

  物品卡必须展示：

  - 图标
  - 名称
  - 尺寸
  - 触发骰点
  - 效果摘要
  - 价格

  验收：1 格到 4 格物品都能表达尺寸差异。

- [ ] **Step 7：创建格子和拖拽组件**

  创建：

  - `装备格 / Equipment Slot`
  - `背包格 / Bag Slot`
  - `已放置物品 / Placed Item Tile`
  - `出售拖放区 / Sell Drop Zone`

  状态：

  - 空格
  - 可放置
  - 不可放置
  - 物品选中
  - 物品拖拽中
  - 战斗触发中

  验收：15 格装备栏在 1440px 画板中可完整展示，背包 2 x 10 结构清晰。

- [ ] **Step 8：创建战斗组件**

  创建：

  - `战斗狗狗面板 / Battle Dog Panel`
  - `生命条 / HP Bar`
  - `骰子显示 / Dice Display`
  - `战斗日志行 / Battle Log Row`
  - `速度控制 / Speed Control`
  - `结果卡 / Result Card`

  验收：战斗中玩家、对手、系统日志三类信息可区分；生命变化不只依赖颜色。

  验收：六种商店的差异通过图形、标题和短文案表达，不需要长规则说明。

  - 匹配按钮

- [ ] **Step 5：创建匹配确认画板**

- [ ] **Step 6：创建战斗回放画板**

  画板名：`PC / 06 战斗回放`

  - 对手装备栏
  - 中央战斗舞台
  - 对手狗狗面板
  - 玩家狗狗面板
  - 中央骰子显示
  - 玩家装备栏

  验收：一眼能看出当前骰点、谁触发了什么物品、双方生命如何变化。

- [ ] **Step 7：创建结算画板**

  画板名：`PC / 07 结算`

  内容：

  - 战斗舞台弱化背景
  - 结果卡
  - 胜负结果
  - 当前胜负记录
  - 积分
  - 继续按钮
  - 重新选择狗狗按钮

  验收：单场战斗结束和整局结束两个操作意图清楚区分。

## 任务 4：补充响应式说明和开发交接标注

**Figma 页面：**

- 修改：`04 Responsive Notes`
- 修改：`03 Screens - PC`

**产出：**

- 移动端适配说明
- 开发交接标注

- [ ] **Step 1：补充移动端布局说明**

  在 `04 Responsive Notes` 中补充 4 个小示意：

  - 顶部状态条纵向堆叠
  - 狗狗图鉴网格变单列或双列
  - 商店工作台纵向分区
  - 装备栏和背包横向滚动

  验收：开发者能从说明判断窄屏布局方向，不需要猜测。

- [ ] **Step 2：标注关键交互状态**

  在相关组件或屏幕旁标注：

  - 金币不足时按钮禁用并显示缺口
  - 拖拽物品时有效格高亮
  - 出售拖放区只在拖拽时强化
  - 战斗触发物品需要高亮
  - 日志默认折叠但可展开

  验收：每个关键状态至少在一个组件或屏幕中可见。

- [ ] **Step 3：标注可扩展狗狗库规则**

  在狗狗图鉴画板旁标注：

  - 初始只有 3 种狗狗，但布局按更多狗狗设计。
  - 卡片网格可滚动。
  - 筛选标签为未来扩展槽位。
  - 详情面板使用选中项数据，不随卡片数量变化。

  验收：后续新增狗狗时不需要重新设计页面结构。

## 任务 5：最终视觉检查

**Figma 页面：**

- 检查全部页面

**产出：**

- 修正后的最终 Figma UI

- [ ] **Step 1：检查风格一致性**

  检查所有画板是否都符合：

  - 手绘粗描边
  - 温暖纸张背景
  - 轻微纸片阴影
  - 活泼但不混乱的颜色
  - 非单一色系

  验收：从任一画板截图都能识别为同一款游戏。

- [ ] **Step 2：检查信息密度**

  重点检查：

  - 商店工作台是否过挤
  - 战斗回放是否能读懂当前事件
  - 狗狗图鉴是否能扩展
  - 结算页主操作是否明确

  验收：核心动作不需要在画板中寻找超过 3 秒。

- [ ] **Step 3：检查文本可读性**

  检查：

  - 所有中文文本没有溢出
  - 按钮文字完整显示
  - 徽章数字清晰
  - 物品名和骰点可读

  验收：缩放到 50% 时仍能分辨主要层级，100% 时正文清晰。

- [ ] **Step 4：导出或截图验证**

  对以下画板做截图检查：

  - `PC / 02 狗狗图鉴`
  - `PC / 04 商店工作台`
  - `PC / 06 战斗回放`
  - `PC / 07 结算`

  验收：截图中无明显重叠、裁切、空白占位误用或风格断裂。
